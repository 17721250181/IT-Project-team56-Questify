IT Project Team56 OOSD-Questify : Testing Summary

Created by Bill Park, last modified by Chih-Yi Huang on Nov 05, 2025

1. Testing Methodology

1.1 Purpose

We verified system reliability with a layered approach: unit → integration → end-to-end → exploratory/manual, to catch defects early, validate interfaces, and confirm real-world user flows in production-like conditions.

2. Test Types and Scope

2.1 Unit Testing

  • Tools: PyTest for the Django backend. (For small, isolated checks on React components we additionally used Vitest, but the primary harness for this project is PyTest.)

  • Scope: Smallest units of logic such as Django models, serializers, utilities, and view logic in isolation.

2.2 Integration Testing

  • Tools: PyTest (with pytest-django).

  • Scope: Interactions between the backend and the database, and backend ↔ external AI API via mocks/fakes. Focused on interface contracts, error propagation, and transaction boundaries.

2.3 End-to-End Testing

  • Approach: Manual testing, did not use the tool Playwright.

  • Scope: Full user journeys: sign in, create a question, receive an answer, run in a realistic environment. We validated navigation, session handling, and expected system responses by following written test scripts.

2.4 Manual / Exploratory Testing

  • Scope: Acceptance checks, usability and responsiveness, accessibility spot checks (keyboard navigation, basic ARIA expectations), and error/edge-case validation beyond scripted paths.

4. Coverage and Quality Assessment

Overall code coverage is 89%, indicating that the critical paths and failure branches in the backend are well exercised. Unit and integration suites both run with PyTest and it provides strong confidence in core logic and service boundaries. The manual end-to-end runs confirm that the main user flows behave as intended in a production-like context.
