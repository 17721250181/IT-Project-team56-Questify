Coding Standards

To ensure maintainability, readability, and consistency across the OOSD Questify System, the following coding standards will be enforced throughout development:

1 General Python/Django Standards

  • Follow PEP 8 for formatting (4-space indentation, 100-character line limit).

  • Organize imports into three blocks: standard library → third-party → local modules.

  • Use type hints and meaningful docstrings (Google or reST style) for all functions and classes.

  • All secrets (e.g., API keys) must be stored in environment variables and loaded via settings.py — never hardcoded.

2 Project Structure

  • Keep a clear separation of concerns:

    • models.py: business rules and data representation.

    • serializers.py: input/output validation and transformation.

    • views.py: thin controllers delegating validation to serializers.

    • urls.py: REST-style endpoints, resource-based (e.g., POST /questions/ instead of /create/).

    • admin.py: register all models with ModelAdmin for visibility.

  • Maintain migrations under migrations/ with descriptive names.

3 Models

  • Use UUIDField for primary keys.

  • Include created_at and updated_at timestamps for all major models.

  • Use TextChoices for enums (e.g., Source, VerifyStatus) to avoid hardcoded strings.

  • __str__ methods must reference existing fields (fixing current bug: Question.__str__ should use question not content).

  • Enforce integrity constraints (e.g., MCQ must always have 5 options and one valid correct answer).

4 Serializers

  • Separate read and create/update serializers (QuestionSerializer vs. QuestionCreateSerializer).

  • Ensure serializer field names match model fields exactly (fixing current mismatch: content → question).

  • Move validation logic (e.g., checking MCQ options) into serializers instead of views.

5 Views & Permissions

  • Keep views thin and delegate validation/business logic to serializers/services.

  • Apply IsAuthenticated for write operations, and AllowAny or IsAuthenticatedOrReadOnly for reads.

  • Standardize response formats:

    • Success: return 201 with resource body.

    • Errors: return structured JSON (e.g., {"error": {"field": "message"}}).

6 API & URL Design

  • Follow REST conventions: resource-based URLs without verbs.

  • Example:

    • POST /api/questions/ → create question.

    • POST /api/questions/{id}/mcq/ → add MCQ details.

    • POST /api/questions/{id}/short/ → add short-answer details.

7 AI Integration

  • OpenAI calls must:

    • Use requests or SDK with a timeout.

    • Be retried up to 2 times with exponential backoff.

    • Fail gracefully with user-friendly messages (e.g., “AI explanation unavailable”).

  • Prompts/templates for AI should be centralized, not hardcoded in views.

8 Testing & CI/CD

  • All new code must have unit tests (PyTest for backend, Vitest for frontend).

  • MCQ/SAQ rules must be explicitly tested (missing options, invalid correct answers).

  • External API calls (OpenAI) must be mocked in tests.

  • GitHub Actions CI will enforce test execution, linting, and coverage (≥80%).
